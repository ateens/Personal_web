요약 현재 구조 가져갈 점 비효율 웹앱 구조 데이터 모델 개발 순서 결정 질문

SYGMA OS를 웹앱으로 옮기기 위한 구조 리뷰

현재 Notion은 `수집 -> 분류 -> 계획 -> 실행/관리 -> 회고/아카이브` 흐름을 linked database view로 구현한 개인 운영체제입니다. 웹앱에서는 이 흐름과 데이터 모델을 보존하고, Notion의 반복 view와 수동 규칙은 제거하는 방향이 적합합니다.

현재 구조

수집 Captures, 빠른 추가, 모바일 입력
분류 Task, Project, Resource, Goal, Box로 정리
계획 미계획, 지연, 이번 주, 이번 달, 타임라인
실행 Today, Tasks, Projects, Resources
회고 Journal, Habits, Obstacles, Archive
DB 역할 웹앱 해석
Captures 빠른 수집함 모든 입력이 처음 도착하는 Inbox
Boxes 삶의 영역과 컨테이너 최상위 life area
Goals 연/분기 목표와 진행률 결과 목표와 성공 기준
Projects 목표를 쪼갠 실행 묶음 기한과 산출물이 있는 project
Tasks 실제 실행 단위 일정, 우선순위, 완료, 시간 기록
Resources 자료, 노트, 스크랩 실행 맥락에 붙는 knowledge base
Bottleneck 장애물과 막힘 blocked reason과 해결 관리
Journals 일간/주간/월간 회고 review log

그대로 가져갈 점

Core

Box -> Goal -> Project -> Task

삶의 영역, 목표, 실행 묶음, 단일 행동으로 내려오는 계층은 유지할 가치가 높습니다.

Workflow

수집과 실행 분리

빠르게 모으고 나중에 분류하는 구조는 개인 관리 앱에서 핵심입니다.

Knowledge

자료를 실행 맥락에 연결

Resource가 목표, 프로젝트, 태스크와 연결되어 자료가 실제 작업에 붙습니다.

Planning

계획하기 작업대

미계획, 지연, 주간 계획, 프로젝트 타임라인을 한 화면에서 다루는 방향이 좋습니다.

Risk

Bottleneck 모델

목표 달성을 막는 문제를 구조적으로 남기는 점은 웹앱에서도 별도 기능으로 유지해야 합니다.

Guides

목표/습관 도우미

긴 템플릿 문서는 단계형 wizard로 바꾸면 더 강력해집니다.

Notion 때문에 생긴 비효율

문제

같은 DB view 반복

홈, 관리 페이지, 상세 템플릿, 모바일 페이지에 동일 DB가 여러 필터로 반복됩니다.

웹앱 대안

데이터 1개, view 여러 개

공통 query layer와 filter preset을 두고 화면별 컴포넌트만 재사용합니다.

문제

DB 간 이동 규칙이 취약

수집함 외 DB끼리 옮기지 말라는 규칙은 Notion 속성 호환성 문제입니다.

웹앱 대안

Capture 변환 플로우

Capture를 원본으로 두고 변환 시 새 엔티티를 생성하며 처리 기록과 undo를 제공합니다.

문제

Task DB 과부하

일반 할 일, 일정, 위임, 나중에, 저널, 습관/루틴이 한 DB에 섞여 있습니다.

웹앱 대안

Habit, Journal 분리

Task는 실행 단위로 두고 반복 습관과 회고는 별도 모델로 분리합니다.

문제

계획시각이 text

정렬, 알림, 시간표, 통계에 불리합니다.

웹앱 대안

날짜 필드 분리

due_at, scheduled_start, scheduled_end, review_at을 분리합니다.

핵심 판단: Notion 화면을 그대로 복제하면 복잡도도 같이 옮겨옵니다. 보존할 것은 데이터 모델과 업무 흐름이고, 버릴 것은 반복 view, 수동 이동 규칙, formula/rollup 의존입니다.

추천 웹앱 구조

화면 목적 핵심 기능
Today 오늘 실행 오늘 할 일, 루틴, 지연 항목, 빠른 수집, 집중 프로젝트
Inbox 수집/분류 빠른 입력, URL 저장, capture 변환, 일괄 분류
Plan 계획 미계획/지연 항목, 주간 캘린더, 프로젝트 타임라인
Tasks 실행 단위 관리 필터, 완료, 일정, 위임, someday/waiting
Projects 프로젝트 관리 상태, 진행률, 타임라인, 연결 task/resource
Goals 목표 관리 연/분기 목표, 진행률, 목표별 프로젝트
Resources 자료/노트 고정, 미분류, 최근, 나중에 보기, 검색
Habits 루틴 반복 규칙, 오늘 체크, streak, 실패/대체 계획
Journal 회고 데일리/위클리/먼슬리 리뷰
Obstacles 막힘 관리 중요한 문제, 사소한 문제, 해결됨
첫 화면은 랜딩 페이지가 아니라 Today가 적합합니다. 상단 빠른 입력, 오늘 할 일, 루틴, 집중 프로젝트, 지연 항목, 오늘 볼 자료가 한 화면에 보여야 합니다.

권장 데이터 모델

핵심 테이블

  • boxes, goals, projects, tasks
  • resources, captures, obstacles
  • habits, habit_instances
  • journals, reviews
  • tags, entity_links, activity_logs

핵심 설계

  • 모든 엔티티는 선택적으로 box_id를 가질 수 있음
  • 자료, 장애물, 태스크는 entity_links로 여러 대상에 연결
  • 반복 습관은 Task 템플릿이 아니라 별도 recurrence 모델
  • 진행률, D-day, 작업 시간은 앱 로직으로 계산

개발 순서

Phase 1

개인 실행 코어

Today, Inbox, Tasks, Boxes, Capture -> Task/Resource 변환, 기본 CRUD.

Phase 2

목표/프로젝트 계층

Goals, Projects, 관계 연결, 프로젝트 상세 내 할 일 관리, 목표별 진행률.

Phase 3

계획 화면

미계획/지연 필터, 주간 캘린더, 프로젝트 타임라인, drag/drop 일정 배치.

Phase 4

자료와 회고

Resources, 에디터, Journal, Weekly/Monthly review, 자료와 작업 연결.

Phase 5

습관과 장애물

Habits, recurrence, habit instances, streak, Obstacles, 막힘 해결 플로우.

Phase 6

가져오기/동기화

Notion export/import, CSV import, Notion API sync, 백업/복원.

우선 결정할 질문

  • 개인 로컬 앱인가, 로그인 기반 클라우드 앱인가?
  • Notion과 계속 동기화할 것인가, 한 번 가져온 뒤 독립 앱으로 갈 것인가?
  • 자료 본문은 Markdown으로 충분한가, block editor가 필요한가?
  • 내부 일정만 관리할 것인가, Google/Apple Calendar 연동이 필요한가?
  • 모바일 입력 속도를 위해 PWA까지 갈 것인가?
